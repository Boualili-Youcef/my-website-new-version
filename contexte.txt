Coffre-Fort Numerique -- Serveur de Paiement Securise
Proof of Concept (PoC) d'un serveur de gestion de donnees bancaires conforme aux exigences du GIE Cartes Bancaires. Le systeme garantit la confidentialite des numeros de cartes stockes grace a un chiffrement multi-acteurs ou aucune personne seule ne peut acceder aux donnees.

Objectifs de Securite
Objectif	Mesure implementee
Confidentialite des donnees	Chiffrement AES-256-GCM (Authenticated Encryption)
Non-persistance de la cle maitre	Stockage exclusif en RAM volatile (/dev/shm)
Double controle obligatoire	Quorum de deux responsables (Technique + Juridique)
Authentification forte	Deux facteurs par acteur (mot de passe + cle USB)
Integrite des donnees	Tag GCM verifie a chaque dechiffrement
Resistance au brute-force	Argon2id (memory-hard KDF, 256 MiB)
Imputabilite	Journalisation SIEM-compatible de chaque operation
Revocation sans partage de secret	Destruction de slots par consensus (Quorum)
Fonctionnalites
Services conformes au cahier des charges
Service	Description	Interface
1.v	Initialisation du coffre (Ceremonie)	setup.py
1.i	Mise en service (Deverrouillage)	CLI / GUI
1.ii	Ajout d'une paire (nom, numero de carte)	CLI / GUI
1.iii	Suppression d'une paire	CLI / GUI
1.iv	Recherche d'un numero de carte	CLI / GUI
1.vi	Revocation d'un acteur (Repudiation)	CLI / GUI
Caracteristiques supplementaires
Architecture Multi-Slots : 4 combinaisons d'acces (2 titulaires + 2 suppleants).
Masquage PAN : les numeros de carte ne sont jamais affiches en entier (PCI-DSS v4.0 3.4).
Protection anti-bruteforce : verrouillage apres 5 echecs consecutifs (GUI).
Effacement securise : zeroization FIPS 140-3 (3 passes) a la deconnexion.
Audit complet : rotation automatique des logs, sanitization anti-injection.
Architecture Generale
setup.py            Ceremonie d'initialisation (enrolement des acteurs)
server.py           Console interactive (CLI)
gui_app.py          Interface graphique (Tkinter)
crypto_utils.py     Module cryptographique central (Argon2id, AES-GCM, Wipe)
audit.py            Journalisation SIEM-compatible
vault.json          Coffre chiffre (genere par setup.py)
cards.db.enc        Base de donnees chiffree (generee a l'usage)
Pre-requis
Python 3.9 ou superieur.
Systeme Linux avec acces a /dev/shm (RAM Disk tmpfs).
WSL est supporte pour les environnements Windows.
Installation
# Cloner le depot
git clone <url-du-depot>
cd ISI_TP03

python3 -m venv venv

# Installer les dependances

Utilisation
1. Initialisation du coffre (une seule fois)
python3 setup.py
Cette commande lance la Ceremonie des Cles :

Generation des fichiers de cle USB pour les 4 acteurs.
Saisie et validation des mots de passe (12 caracteres minimum, complexite imposee).
Creation du fichier vault.json contenant les 4 slots chiffres.
Les dossiers usb_tech_main/, usb_tech_rep/, usb_jur_main/, usb_jur_rep/ simulent les supports amovibles. En production, ces fichiers doivent etre stockes sur des cles USB physiques conservees dans un coffre-fort.

2. Demarrage du serveur
Mode console (CLI)
python3 server.py
Mode interface graphique
python3 gui_app.py
3. Deverrouillage
Deux responsables doivent s'authentifier simultanement :

Selectionner le role (Titulaire ou Suppleant) pour chaque domaine.
Fournir le chemin vers le fichier de cle USB (.bin).
Saisir le mot de passe.
4. Operations courantes
Une fois le coffre deverrouille :

Ajouter : saisir un nom et un numero de carte (13 a 19 chiffres).
Rechercher : saisir un nom pour afficher le numero masque.
Supprimer : saisir un nom pour retirer l'enregistrement.
5. Revocation
La destruction des droits d'un acteur se fait par consensus :

Un binome valide doit s'authentifier.
L'acteur cible est selectionne.
Les slots impliquant cet acteur sont detruits.
Double confirmation requise (GUI).

Limitations connues
Limitation	Justification
Base de donnees en dict Python (nom = cle)	PoC — en production, utiliser un SGBD avec cle primaire UUID.
Cles USB simulees par des dossiers locaux	PoC — en production, utiliser de vrais supports amovibles chiffres.
Pas de gestion de sessions concurrentes	Mono-utilisateur par conception (acces physique requis).
Pas de HSM materiel	PoC — en production, deleguer la gestion des cles a un HSM certifie.
Effacement memoire best-effort	Python ne garantit pas la zeroization via le garbage collector.
Revocation simplifiee (suppression de slots)	PoC — le rapport decrit une regeneration complete non implementee.

Security Audit

Lab Kubernetes DevSecOps montrant pourquoi RBAC ne suffit pas, puis ajoutant progressivement du policy-as-code, du contrôle d'admission, de la détection runtime et du confinement réseau.

Problème
Kubernetes RBAC répond à une question précise : qui peut appeler quelle API ? Il ne répond pas à la question suivante : est-ce que le contenu du workload envoyé au cluster est sûr ?

Ce lab démontre qu'une identité développeur limitée peut ne pas avoir le droit de lire les secrets, tout en ayant encore le droit de créer des pods dangereux, par exemple :

un conteneur privileged
un montage hostPath sur le système de fichiers du nœud
un token de ServiceAccount monté automatiquement dans le pod
Le projet construit ensuite une chaîne de défense autour de ces risques : régression policy-as-code en CI, contrôle d'admission avec Gatekeeper et Kyverno, détection runtime avec Falco, puis confinement réseau ciblé avec NetworkPolicy.

Architecture

Loading
Ce schéma montre la séparation importante du projet :

la CI exécute une régression : les manifests d'attaque doivent être refusés et les manifests sûrs doivent passer
la CI échoue uniquement si ce comportement attendu n'est plus respecté
RBAC autorise ou refuse l'appel API
Gatekeeper ou Kyverno contrôlent le contenu du manifest à l'admission
Falco observe le comportement après le démarrage du pod
NetworkPolicy limite certains flux réseau après la création du pod
Couches de défense
Couche	Outils	Ce que le lab prouve
Autorisation	Kubernetes RBAC	dev-user ne peut pas lire les secrets, mais peut créer des pods
Analyse statique	Conftest / OPA	les manifests dangereux sont détectés avant le cluster
Admission	Gatekeeper et Kyverno	les workloads dangereux sont rejetés par l'API Kubernetes
Détection runtime	Falco	les exécutions de curl et sh sont détectées après admission
Confinement réseau	NetworkPolicy	un pod conforme peut démarrer tout en ayant son egress bloqué

Ce que le lab démontre
RBAC et admission control répondent à deux problèmes différents.
Les contrôles policy-as-code en CI réduisent les manifests risqués avant le déploiement.
L'admission reste nécessaire, car une CI peut être contournée par un kubectl apply direct.
Le comportement runtime n'est pas entièrement visible depuis un contrôleur d'admission.
Falco ajoute de la visibilité runtime, mais il détecte au lieu de bloquer automatiquement.
NetworkPolicy ajoute une première couche de confinement réseau après la création du workload.
Le scan de charts Helm montre que la même logique de politiques peut être réutilisée côté supply chain.
Limites connues
Le lab cible un cluster Kind local, pas une plateforme Kubernetes de production.
La preuve NetworkPolicy est ciblée sur un pod de démonstration, pas sur une segmentation complète de namespace.
Falco détecte les comportements runtime mais ne bloque pas automatiquement leur exécution.
GitHub Actions valide le contenu du dépôt, mais le déploiement reste local car le runner ne peut pas atteindre le cluster Kind.
Les politiques couvrent les scénarios du lab, pas l'intégralité du CIS Kubernetes Benchmark.
Le fichier dev-user-kubeconfig.yaml est généré localement et ne doit pas être versionné.

CloudSecVision
License Python AWS Streamlit Ollama

CloudSecVision is a Python-based AWS security scanner with a Streamlit dashboard and optional local AI analysis through Ollama. It helps review common IAM, S3, and EC2 misconfigurations and writes scan results as local JSON reports.

Features
IAM policy review for overly permissive permissions, wildcard actions, risky managed policies, and stale access keys.
S3 bucket review for public access, bucket policies, ACL exposure, encryption, versioning, logging, and lifecycle configuration.
EC2 security group review for public inbound access, sensitive ports, broad port ranges, default security groups, and unrestricted outbound rules.
Streamlit dashboard for launching scans and reviewing summarized findings.
Optional AI-generated analysis using a local Ollama server.
Docker and Docker Compose support for local deployment.
Docusaurus documentation site under website/.
Requirements
Python 3.8 or newer.
AWS credentials configured locally with read permissions for IAM, S3, EC2, SSM, ELB, and ELBv2 checks.
Optional: Docker and Docker Compose.
Optional: Ollama for AI analysis.
Recommended AWS setup:

aws configure
Use least-privilege read-only permissions where possible. Do not commit AWS credentials, generated reports, or local virtual environments.

Quick Start
git clone https://github.com/Boualili-Youcef/CloudSecVision.git
cd CloudSecVision

python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

./run_dashboard.sh
Dashboard URL:

http://localhost:8501
Command Line Usage
Run all scanners:

python main.py
Run one service scanner:

python main.py --service iam
python main.py --service s3
python main.py --service ec2
Run all scanners with AI analysis enabled:

python main.py --service all --ai
Supported services are iam, s3, ec2, and all.

Dashboard
Start the dashboard:

./run_dashboard.sh
The dashboard reads generated JSON reports from scan/results/ when they exist and can launch new scans from the UI.

AI Analysis
AI analysis is optional and uses Ollama at http://localhost:11434.

Start Ollama with Docker:

docker run -d \
  -p 11434:11434 \
  -v ollama_data:/root/.ollama \
  --name cloudsecvision-ollama \
  ollama/ollama:latest
Install the model expected by the analyzer:

docker exec cloudsecvision-ollama ollama pull llama3.2:3b
Then run:

python main.py --service all --ai
Docker Usage
Build and run the main application:

docker build -t cloudsecvision:latest .
docker run -d \
  -p 8501:8501 \
  -v ~/.aws:/home/cloudsec/.aws:ro \
  --name cloudsecvision-app \
  cloudsecvision:latest
Or use Docker Compose:

docker compose up --build cloudsecvision
The compose file also defines optional ollama and docs services.

Documentation Site
The documentation site is in website/.

cd website
npm install
npm start
Documentation URL:

http://localhost:3000/cloudsecvision/
Security Notes
Scan results can contain AWS resource names, ARNs, account details, and configuration data. They are ignored by Git by default.
venv/, node_modules/, Docusaurus build output, AWS credentials, and generated reports should stay out of version control.
The repository history has been cleaned to remove a previously committed AWS CLI archive and virtual environment.
After rewriting Git history, publish the cleaned branch with a reviewed --force-with-lease push and coordinate it with anyone else using the repository.
